Rxjava2.x的基本概念和使用

本文主要记录Rxjava2.0的基本使用，详细的推荐https://www.jianshu.com/u/c50b715ccaeb,水管概念。

官方github地址https://github.com/ReactiveX/RxJava，

Rxjava 是什么

Rxjava在github上的自我介绍是”RxJava – Reactive Extensions for the JVM – a library for composing asynchronous and event-based programs using observable sequences for the Java VM.”(一个使用Java VM的可观察序列来组合异步和基于事件的程序的库)。简而言之就是一个异步+链式操作的库,而其异步实现是通过一种扩展的观察者模式来实现的。

也就是说Rxjava是一种Java语言实现的响应式编程，是创建基于事件的异步程序，是以数据流为核心，处理数据的输入，处理以及输出的。

观察者模式

Rxjava基于观察者模式，主要有四个部分被观察者、观察者，订阅、事件。有两种观察者模式：

  • Observable ( 被观察者 ) / Observer ( 观察者 )
  • Flowable （被观察者）/ Subscriber （观察者）
    在RxJava2.x中，Observable用于订阅Observer，不再支持背压(1.x中可以使用背压策略)，而Flowable用于订阅Subscriber，是支持背压(Backpressure)的。
Observable/Flowable使用场景

何时使用Observable：

  • 最长的流量不超过1000个元素：即，随着时间的推移，元素很少，因此在您的应用程序中几乎没有机会使用OOME。
  • 您处理GUI事件，例如鼠标移动或触摸事件：这些事件很少能够合理地反压并且不常见。您可以使用Observable处理1000 Hz或更低的元素频率，但无论如何都要考虑使用采样/去抖动。
  • 您的流程基本上是同步的，但您的平台不支持Java Streams，或者您错过了它的功能。使用Observable通常比开销更低Flowable。（您还可以考虑IxJava，它针对支持Java 6+的Iterable流进行了优化）。

何时使用Flowable

  • 处理以某种方式在某处生成的10k +元素，因此链可以告诉源限制它产生的量。
  • 从磁盘读取（解析）文件本身就是阻塞和基于拉取的，当你控制时，它可以很好地处理背压，例如，你从指定的请求数量中读取了多少行。
  • 通过JDBC从数据库读取也是阻塞和基于拉取的，并且可以通过调用ResultSet.next()可能的每个下游请求来控制。
  • 网络（流）IO，其中网络帮助或使用的协议支持请求一些逻辑量。
  • 许多阻塞和/或基于拉的数据源，最终可能会在未来获得非阻塞的反应API /驱动程序。

背压Backpressure

所谓背压就是生产者（被观察者）的生产速度大于消费者（观察者）消费速度从而导致的问题。

当上下游在不同线程时，上游发送速度比下游处理速度快，最终会导致数据积累越来越多，从而导致OOM等异常，这便是响应式编程中的背压（backpressure）问题。Rxjava中我们可以使用Flowable/Subscriber来解决背压问题，让下游控制上游的发送速度，响应式拉取。

  • 上游可以通过emitter.requested()获取异步缓存池中可放入数据的数量，上游通过Subscription的request()设置请求事件数量。
  • Flowable缓存默认128个，
  • 重复调用是增加可处理事件总数，调用必须在同一线程；
  • 上下游不同线程，设置request请求无法改变上游，上游的requested的值内部调用会在合适的时候自动触发（消费96了事件后），不过上游可以通过当前的requested的值是否大于0，若等于0则说明下游处理不过来了，则需要等待。

BackpressureStrategy模式：

  • BackpressureStrategy.MISSING:
  • BackpressureStrategy.ERROR: MissingBackpressureException,当下游处理速度跟不上
  • BackpressureStrategy.BUFFER:宽展缓存区，无限大，还是可能OOM
  • BackpressureStrategy.DROP:把存不下的事件直接丢掉
  • BackpressureStrategy.LATEST:只保留最新的事件覆盖掉老事件（最后一个事件不停被覆盖）

官方对背压的介绍https://github.com/ReactiveX/RxJava/wiki/Backpressure

线程切换

subscribeOn(Scheduler scheduler)：指定上游发送时间的线程，只有第一次有效。
observeOn(Scheduler scheduler)：指定下游接收事件的线程，可以多次调用，每调用一次下游线程切换一次。

Scheduler种类：

  • Schedulers.single():拥有一个线程单例，所有的任务都在这一个线程中执行，当此线程中有任务执行时，其他任务将会按照先进先出的顺序依次执行。
  • Schedulers.computation():用于CPU 密集型计算任务。
  • Schedulers.io():用于IO密集型的操作，例如读写SD卡文件，查询数据库，访问网络等，具有线程缓存机制。
  • Schedulers.trampoline():在当前线程立即执行任务，如果当前线程有任务在执行，则会将其暂停，等插入进来的任务执行完之后，再将未完成的任务接着执行。
  • Schedulers.newThread():在每执行一个任务时创建一个新的线程，不具有线程缓存机制。
    这些可以在所有JVM平台，在Android上有定义地的：
  • AndroidSchedulers.mainThread():在Android UI线程中国之行任务，需要引入rxandroid包。
    此外，可以通过Scheduler .from(Executor)将现有的Executor(及其子类型，如ExecutorService)包装到调度器中：
  • Scheduler.from(@NonNull Executor executor)：指定一个线程调度器，由此调度器来控制任务的执行策略。

其他

Single和SingleObserver

如果你使用一个单一连续事件流，即只有一个onNext事件，接着就触发onComplete或者onError，可以使用Single。
Single只包含两个事件，一个是正常处理成功的onSuccess，另一个是处理失败的onError，它只发送一次消息没有onNext()方法，有onSucces()方法。

Completable和CompletableObserver

如果你的观察者连onNext事件都不关心，你可以使用Completable，只有onComplete和onError两个事件。

Maybe和MaybeObserver

如果你有一个需求是可能发送一个数据或者不会发送任何数据，这时候你就可以使用Maybe，它类似于Single和Completable的混合体。

Maybe可能会调用以下其中一种情况（也就是所谓的Maybe）：

  • onSuccess或者onError
  • onComplete或者onError
    onSuccess和onComplete是互斥的存在
RxBus

当用Rxjava来替代EventBus之类的事件通知时，如无背压可以使用PublishSubject.create()生成Subject对象，既当观察者又当被观察者；有背压则可以使用PiblishProcessor。
